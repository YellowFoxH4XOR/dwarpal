# Automated code review and rule-conformance checking: what exists, and where the novel opening is

**Scope.** This review targets one specific idea: a system where a team authors a set of rules once, every pull request is checked against that same rule set, and any deviation is flagged on the offending commit. That sits at the meeting point of three literatures that have historically been separate — empirical *modern code review* (what human reviewers actually do), *static analysis / linting* (deterministic rule enforcement), and the fast-moving *LLM-based code review* line (2022–2026). The interesting finding is that the field is converging on your exact framing from two directions at once, but no published system yet delivers the specific combination you're describing. That gap is the novelty.

## What code review is actually for

The empirical anchor for this whole area is [Bacchelli & Bird 2013](https://doi.org/10.1109/icse.2013.6606617), the "Expectations, outcomes, and challenges of modern code review" study at Microsoft. Its central result reframes what a reviewer is for: developers *expect* code review to find defects, but its measured outcomes are dominated by knowledge transfer, team awareness, and — crucially for you — maintaining consistency with team conventions and design intent. Defect-finding is real but secondary and unreliable, which matters because a rule-conformance reviewer is not a weaker bug-finder; it directly automates the convention-and-consistency function that human reviewers spend much of their attention on and do inconsistently; the gap it fills is documented, not invented.

The counterpoint from the same literature is that consistency enforcement is exactly where humans are worst: reviewers apply style and convention rules unevenly, miss violations under time pressure, and disagree with each other. That variance is the argument *for* mechanizing rule conformance, and the argument *against* leaving it to an LLM that will be just as inconsistent unless the rules are pinned down explicitly.

## The deterministic pole: linters and SAST

The mature, deployed answer to "flag every deviation from a rule" is static analysis. The reference account of making this work in practice is Google's, across [Sadowski et al. 2015 (Tricorder)](https://doi.org/10.1109/icse.2015.76) and its retrospective [Sadowski et al. 2018](https://doi.org/10.1145/3188720). Their hard-won lessons are the design constraints your system inherits regardless of implementation: analyses must run *at review time* inside the code-review tool (not as a separate batch), the false-positive rate is the single variable that determines whether developers keep the tool on, results must be *actionable* with a suggested fix, and rule sets must be extensible by the teams that own the code rather than by a central authority. Google found that once the effective-false-positive rate crept above roughly one in ten, developers stopped trusting and eventually disabled checks, a number worth treating as a hard budget for any deviation-flagging system.

The limit of the deterministic pole is expressiveness. Rule engines and SAST tools enforce what can be written as an AST pattern or dataflow query; a recent head-to-head evaluation of Java SAST tools ([Li et al. 2023](https://doi.org/10.1145/3611643.3616262)) documents both their incomplete recall and their noise. Rules that require understanding intent ("error messages must name the failing resource," "new public APIs must match the naming pattern of the surrounding module," "this change should update the changelog") fall outside what a linter can express. That expressiveness gap is precisely the seam the LLM work is now prying open.

## The learned pole: convention models to LLM agents

Before LLMs, the learned-model line already targeted conventions directly: [Pradel & Sen 2018 (DeepBugs)](https://doi.org/10.1145/3276517) framed a class of bugs as *deviations from statistical naming and usage conventions* learned from a corpus, conceptually the ancestor of "flag the commit that departs from how this codebase does things." The modern-transformer turn came with [Tufano et al. 2021](https://doi.org/10.1109/icse43902.2021.00027) and, most influentially, [Li et al. 2022 (CodeReviewer)](https://doi.org/10.1145/3540250.3549081), which pre-trained a model specifically on review activities — predicting review necessity, generating review comments, and refining code from comments — establishing the task suite the field still evaluates against. The general code-LLM substrate under all of this is [Chen et al. 2021 (Codex)](https://doi.org/10.48550/arXiv.2107.03374).

The 2024–2026 wave moves from single models to **agents**. [CodeAgent (Zhang et al. 2024)](https://doi.org/10.48550/arXiv.2402.02172) is the most relevant single paper to your idea: a multi-agent reviewer whose evaluated tasks explicitly include *validating code-style adherence* and *detecting inconsistencies between code changes and commit messages*, with a supervisory "QA-Checker" agent that keeps the other agents on the original review question. That supervisory-consistency design directly answers to the LLM-variance problem raised above. The enterprise-deployment counterpart is [RovoDev Code Reviewer (Atlassian, 2026)](https://doi.org/10.1145/3786583.3786851), a year-long online evaluation of a review-guided, context-aware, quality-checked comment generator built *without* fine-tuning and wired into Bitbucket. It is the closest published thing to a production PR-gate reviewer, and evidence that the deployable design point is prompt-and-context engineering over a frontier model rather than a bespoke trained model.

## The convergence point — and the gap

Two very recent papers sit almost exactly on your concept and are worth reading closely. [LintCFG (2026)](https://doi.org/10.48550/arXiv.2602.07783) proposes a domain-specific language for expressing coding rules in a tool-agnostic way and uses an LLM to *compile* those rules into concrete linter configurations — treating "author the rules once, enforce everywhere" as a compiler problem. [Checkstyle+ (2025)](https://doi.org/10.48550/arXiv.2510.23068) is the hybrid you'd likely converge on: it runs a deterministic linter (Checkstyle) for the rules static analysis handles cleanly, and delegates *only the subset of rules requiring semantic understanding* to an LLM. Together these two papers establish the emerging consensus architecture — **a rule specification layer, a deterministic enforcement layer, and an LLM layer reserved for the rules the deterministic layer cannot express**, very likely the right skeleton for what you're building.

What none of them delivers, and where the opening is:

1. **Rules as a first-class, versioned, testable spec that governs the whole pipeline.** LintCFG compiles rules *into linter configs*; Checkstyle+ hard-codes which rules go to the LLM. Nobody treats the rule set itself as a maintained artifact with its own tests, a coverage metric ("which of our rules actually fire on real PRs?"), and per-rule precision tracking. A system where each rule carries a small suite of positive/negative example diffs — so you can measure and regression-test the reviewer's judgment on *your* rules, is absent from the published literature and directly attacks the false-positive-budget problem that Google's work says is decisive.

2. **Deviation attribution at commit/hunk granularity.** The agent papers ([CodeAgent](https://doi.org/10.48550/arXiv.2402.02172), [RovoDev](https://doi.org/10.1145/3786583.3786851)) generate PR-level review comments. Your framing, *flag the specific commit that deviates*, is a blame/attribution problem the review literature hasn't formalized: mapping a rule violation back to the minimal introducing change rather than the whole PR.

3. **Calibrated abstention over recall.** The newest evaluation work, [CR-Bench (2026)](https://doi.org/10.48550/arXiv.2603.11078), reports the field's central failure mode bluntly: agents tuned to catch *every* hidden issue produce a low signal-to-noise ratio that obscures real progress and burns developer trust, the modern restatement of Google's false-positive lesson. A rule-conformance reviewer that is *allowed to say "no confident violation"* per rule, and is measured on precision-at-flag rather than recall, is the design most of these papers implicitly want but don't build.

4. **Benchmarking is still forming, which is an opportunity.** [CR-Bench](https://doi.org/10.48550/arXiv.2603.11078) and the concurrent c-CRAB benchmark ([Code Review Agent Benchmark, 2026](https://doi.org/10.48550/arXiv.2603.23448)) only appeared in 2026, and both evaluate *general* review quality rather than *rule-conformance* specifically. There is no standard benchmark for "given this rule set and these PRs, did the system flag the right deviations on the right commits." Publishing that benchmark alongside the system is a credible, defensible contribution on its own.

The vision papers agree the moment is right: [Rethinking Code Review in the Age of AI (2026)](https://doi.org/10.48550/arXiv.2605.17548) argues that AI code generation is inflating review volume faster than review capacity, that current tools are fragmented across isolated sub-tasks, and that the effective unit is the *end-to-end PR lifecycle with context carried across stages*, the umbrella your rule-conformance gate fits under.

## Bottom line for building something novel

The defensible, unoccupied position is not "an LLM that reviews PRs" — that space is crowded and, per Atlassian's deployment, increasingly a matter of engineering rather than research. The opening is the **rule-centric** reframing: treat the team's rule set as a versioned, example-tested specification; route each rule to the cheapest layer that can enforce it (deterministic where possible, LLM only where semantics are required, following [Checkstyle+](https://doi.org/10.48550/arXiv.2510.23068)); attribute every flagged deviation to the minimal introducing commit; optimize for precision-at-flag with per-rule calibrated abstention against an explicit false-positive budget (following the lesson from [Google's static-analysis experience](https://doi.org/10.1145/3188720) and [CR-Bench](https://doi.org/10.48550/arXiv.2603.11078)); and ship a rule-conformance benchmark that doesn't yet exist. Each of those four is individually a paper-sized contribution, and together they describe a system materially different from everything currently published.
